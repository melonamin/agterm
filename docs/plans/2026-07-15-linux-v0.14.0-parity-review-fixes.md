# Plan: Linux v0.14.0 parity, review fixes, and verified release

Bring `linux-port` to the exact upstream `v0.14.0` release, close every defect found in the deep Linux-port review,
and publish a verified `linux-v0.14.0` release from `linux-port`.
The implementation must preserve agterm as one product: shared behavior stays portable and upstream-compatible,
while GTK/libadwaita, Linux process handling, libghostty FFI, desktop integration, packaging, and release mechanics
stay in Linux-owned targets.

The selected upstream boundary is the immutable `v0.14.0` tag at `ed0e4c6`.
Do not roll unreleased upstream work into this release.
At plan-writing time `upstream/master` has two known post-tag commits, `8496d9e` and `06a5dff`; fetch again during
execution and exclude those plus any later post-tag commits from both `master` and `linux-v0.14.0`.

The terminal state is a public GitHub release named `linux-v0.14.0` whose tag points to the validated candidate on
`origin/linux-port`, with tar, DEB, RPM, AppImage, checksums, and attestations all present and independently checked.
The release payload must contain the pinned Ghostty theme catalog and a working `xterm-ghostty` terminfo entry.

The following constraints apply to every task:

- `agtermCore/Sources/agtermCore/` must not import or call GTK, libadwaita, AppKit, SwiftUI, Metal, GhosttyKit,
  libghostty FFI, Glibc, CoreGraphics, or another host runtime.
- Any shared-package delta must use portable data and be suitable for upstream agterm.
  Prepare such work independently from Linux-only changes.
- Never make a downstream edit to
  `agtermCore/Tests/agtermCoreTests/ConfigPathsTests.swift`.
- Accept upstream's `v0.14.0` `CHANGELOG.md` verbatim during the merge, but do not add downstream release notes to it.
- Keep Linux-specific behavior in `agterm-linux/`, `linux/`, `packaging/linux/`, Linux workflows, or Linux scripts.
- Preserve existing control commands and read-back.
  Dashboard chrome and notification-click routing are explicitly control-API exempt because they compose existing
  `dashboard`, `window.select`, and `session.select` behavior rather than introduce a new state mutation.
- Preserve the existing Linux Dock-bounce exemption: Linux desktop environments expose no portable equivalent of
  macOS Dock attention requests.
- Keep commits narrow: upstream synchronization, portable seams, each safety fix, parity features, CI, and release
  preparation must remain reviewable independently.
- Stop for explicit approval immediately before pushing `master` or `linux-port`, publishing an upstream PR branch,
  creating or pushing `linux-v0.14.0`, moving a remote tag, or publishing the final release text.
