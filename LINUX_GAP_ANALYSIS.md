# agterm Linux/macOS UI Parity & Core-Hoist Gap Analysis

agterm's strategy is a **shared business logic + controller in `agtermCore`** with a **thin, platform-specific UI** on top — macOS SwiftUI/AppKit and Linux GTK4/Adwaita driving the same model. Today that split is uneven: the macOS UI is ~10.4k LOC, the Linux UI ~2.0k LOC, both over a 31-file shared core — but a large amount of *controller* work (control dispatch, settings lifecycle, palette catalog, keymap resolution, notification policy, split-pane focus) still lives in the macOS **app target**, so Linux must re-implement it and immediately drifts. This document is a backlog ordered to make the **architecture placement** (logic to hoist into `agtermCore`) first-class, because that is what makes the thin-UI strategy real rather than two diverging forks.

Three dimensions sit *outside* that hoist axis and were previously invisible: the libghostty **action-callback layer** (treated as "rendering parity ✅" when it is not), the **Linux OS-integration layer** (packaging, activation, display-server matrix), and **testing/CI** (none exists on Linux). They are now first-class sections; the agtermCore-hoist work remains the priority.

## Executive summary

After applying the verifier corrections (Linux already has a basic theme picker, a search bar, `SettingsStore` usage, `theme.set`/`theme.list`/`config.reload` arms, and — re-baselined against the current working tree — the full `WindowLibrary` multi-window adoption with `window.*` control; these are **partial/done**, not missing), merging duplicate rows, and adding the action-callback / platform / testing dimensions, the findings come to **≈105 actionable items** plus ~24 confirmed-matched parity rows (no action).

| Bucket | Count |
|---|---|
| **By priority** | P0 = 4 · P1 ≈ 27 · P2 ≈ 58 · P3 ≈ 16 |
| **By gap type** | architecture-placement ≈ 37 · missing-feature ≈ 42 · behaviour-delta/bug ≈ 26 |
| **Matched (no action)** | ≈ 24 parity rows (snapshot/restore, fuzzy scorer, scroll/HiDPI content-scale, nav scoping, wire protocol, CLI tree, primary-selection clipboard, WindowLibrary ownership) |

**Headline gaps**

1. **The control channel is forked (structurally — behavioral drifts now resolved).** Linux's `AppController.handleControl` re-implements the macOS `ControlServer.dispatch` switch. The three behavioral drifts are FIXED: mode parsing now rejects unknown modes everywhere (no silent toggle), error strings are de-forked via shared `ControlResolve.notFoundMessage`/`validateSessionNewAddressing`, and `restore.clear` is wired. **All 46** commands are wired + runtime-verified. The remaining work is the STRUCTURAL unification into a shared `ControlDispatcher` (so the two servers become thin transports) — the big-bang refactor, not a behavioral gap.
2. **There is no settings controller.** ~80% of macOS `SettingsModel` is UI-agnostic Foundation logic stranded in the app target. Linux loads `SettingsStore` only to emit the theme line and **drops** font-family/font-size/background-opacity/mouse-scroll-multiplier that the shared `AppSettings.ghosttyConfigLines()` already produces — and there is no Settings window.
3. **Split panes are half-modeled.** `session.splitFocused` is never written on Linux, so the sidebar row name, window title, `focusedCwd`, `session.type`/`session.copy`, and in-terminal search all stay pinned to the primary pane; the split ratio is never persisted; and a primary-shell exit destroys the *whole* session including the live split shell.
4. **The keymap layer is bypassed.** Linux hardcodes GDK keyvals in the surface input handler, so shortcuts aren't rebindable, `keymap.conf` is never read, and custom shell-command bindings don't exist at all.
5. **The sidebar is feature-thin.** No right-click context menu, no unseen badge, no drag-reorder, and a full teardown/rebuild on every mutation; workspace move/delete/focus/rename are reachable only over the socket.
6. **Notifications are a bare `notify-send`.** No focus-suppression, no unseen-badge bump, no click-to-reveal identity, no coalescing — the entire orchestration layer the macOS `NotificationManager` owns is absent. The audible/visual **bell** (`RING_BELL`) is wired on neither platform.
7. **Multi-window has landed but is unfinished.** Linux now adopts `agtermCore.WindowLibrary` (per-window `AppStore`, frontmost routing, reopen-on-launch, `window.new/list/select/close/rename/delete`). Remaining: no GUI window menu, no `window.resize`/`move`, no quit confirmation, no quit-flush of the latest cwd, no per-window geometry restore, no quick terminal.
8. **The host action-callback layer is below parity.** "Rendering is parity ✅" hides it: macOS `GhosttyCallbacks.action` handles 11 `GHOSTTY_ACTION_*` tags, Linux `GhosttyApp.handleAction` handles 9. Mouse-hide-while-typing (`MOUSE_VISIBILITY`) and font-size persist (`CELL_SIZE`) are macOS-only; bell, URL-open, link-hover cursor, and progress-report are handled on neither. Plus per-display binding (`ghostty_surface_set_display_id`) is macOS-only, so multi-monitor HiDPI can mis-scale on Linux.
9. **There is no Linux platform/packaging layer.** No `.desktop` entry, no app icon, no flatpak/AppImage/tarball, no app install target (`packaging/` holds only the macOS `agterm.rb`); `GApplication` runs with `GApplicationFlags(rawValue: 0)` and no decided single-instance/activation routing; no X11-vs-Wayland matrix and no GL-context fallback.
10. **There is no Linux test or CI dimension.** `agterm-linux` has no `Tests` target, there is no GTK UI-test harness, and `ci.yml` builds/tests only on `macos-26` — the forked `handleControl` switch has zero round-trip/e2e coverage, violating the repo's keep-in-sync test rule.
11. **Ghostty resources aren't shipped.** No bundled terminfo/shell-integration and a forced `TERM=xterm-256color`, so `xterm-ghostty` capabilities, OSC 7 cwd / OSC 133 prompt marks, and the agent-status hooks degrade on a clean box.

**The single biggest architectural theme:** *controller logic is sitting in the macOS UI target.* Most Linux gaps are downstream of one pattern — orchestration that should be platform-agnostic lives in SwiftUI/AppKit, forcing Linux to duplicate it. The fix is to **hoist that logic into `agtermCore` behind small platform-applier protocols**, leaving each UI as a thin *translate-input → call core → reconcile* adapter. Section 4 is therefore the priority. The action-callback, platform-integration, and testing sections capture the gaps that no hoist row touches.

## Architecture: logic to move into agtermCore

This is the priority work. Each hoist below removes a place where Linux would otherwise duplicate macOS logic; the leading clusters unblock the most downstream features. Where a hoist introduces debounce, **back it with the already-shipped host-free `agtermCore.Debouncer`, not a GTK `g_timeout`** — reusing core is the whole thesis.

## Linux platform integration

These have **no macOS equivalent** and fall outside the hoist axis — they are net-new Linux platform work. Where core logic helps it is noted.

### Packaging & desktop integration
- **No `.desktop` entry** → the app can't be launched from the desktop menu, has no taskbar/notification icon association, and can't register URL/MIME handlers. (S, **P1**).
- **No installed app icon** — `AppIcon.icon` is Icon-Composer/macOS-only; the `GApplication` id is `com.umputun.agterm.linux` with no themed icon shipped to `hicolor`. (S, **P1**, pairs with `.desktop`).
- **No flatpak / AppImage / tarball / distro package and no app install target** — `packaging/` holds only the macOS `agterm.rb`; `LINUX_PORT.md` explicitly defers distribution to "M5", but it was never listed as work here. Pick one initial channel (flatpak is ghostty's choice) + a `just`/script install target (M, **P2**).

### App lifecycle — single-instance / activation
- `App.main` constructs the `AdwApplication` with `GApplicationFlags(rawValue: 0)` (default single-instance, no `HANDLES_OPEN`/`COMMAND_LINE`/`SERVICE`) and bootstraps the library once in `onActivate`. There is **no decided routing** for a second `agterm` launch (or a `.desktop` launch while running): does it remote-activate the running `GApplication` and open/focus a window, or defer to the control socket? There is no analog of the macOS `application(open:)` / reopen-in-new-window path. Multi-window and "Open directory in new window" depend on this. → decide and implement the activation contract (resolution can reuse `ControlResolve`/`WindowLibrary`) (M, **P2**).

### Display-server matrix (X11 vs Wayland) & GL robustness
- `GhosttySurface.realize` hard-requires an OpenGL context (`GtkGLArea`, `GDK_GL_API_GL`); on `gtk_gl_area_get_error` it only writes to stderr with **no fallback** — a clean-box/VM/headless/llvmpipe or Wayland-without-GL session can get a blank or non-starting terminal (Metal on macOS has no silent-fail equivalent). Add a user-visible error surface + software-render path (M, **P2**).
- **No X11-vs-Wayland matrix** is tracked: primary-selection (wired ✅, but X11/Wayland differ), CSD vs SSD window decorations, IME/compose, and clipboard/`GDK_BACKEND` behaviour all vary by backend. Enumerate and test both backends (M, **P2**).

### Accessibility / automation identifiers
- macOS SwiftUI views carry `accessibilityIdentifier` hooks (palette/settings) that XCUITest drives; GTK widgets set **no AT-SPI accessible names or automation ids**, and terminal content is a11y-opaque on both (`GtkGLArea`/`NSView` expose no screen-reader text). Without accessible names the future GTK UI tests have nothing to target (this compounds the testing gap). → set `gtk_accessible`/`accessible-name` on the sidebar/palette/settings widgets (S, **P2**).
